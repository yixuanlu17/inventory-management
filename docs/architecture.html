Architecture overview

Factory Inventory Management System

A three-tier demo application: a Vue 3 single-page client, a FastAPI HTTP backend, and JSON-backed in-memory mock data. No database, no auth — built for fast iteration on inventory, orders, demand, and spending workflows.

1 System Architecture

Each tier is independently runnable. The browser talks to the API over HTTP/JSON; the API loads JSON files into memory once at startup and filters in Python on every request.

5 Key Patterns

Filter system

Four filters — time period, warehouse, category, order status — are managed in the useFilters composable and applied uniformly as query params across endpoints.

Reactivity split

Raw API responses live in ref()s; every derived value (sorted lists, totals, chart data) is a computed() so the cache stays correct on filter change.

Quarter mapping

The month filter accepts either a literal YYYY-MM or a quarter id (Q1-2025…), which expands to its three months via QUARTER_MAP.

Pydantic at the edge

Every endpoint declares a response_model. JSON files are the source of truth; if data shape drifts, validation fails fast at the response boundary.

Stateless backend

Data is loaded once at process start and never persisted back. POSTed purchase orders live only until restart — perfect for demos, unsafe for prod.

Open CORS

allow_origins=["*"] by design so the Vite dev server on :3000 can hit the API on :8001. Restrict this before exposing the backend anywhere real.
